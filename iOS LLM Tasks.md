This ecosystem already has routing (`ProviderGatewayKit`), structured decoding (`StructuredOutputKit`), cost metering (`TokenMeterKit`), caching (`ResponseCacheKit`), and schema-validated tool dispatch (`ToolRegistryKit`) — but nothing that actually *drives* a multi-step tool-calling loop for a plain-text (non-tool-calling) model. That gap matches the "agent-loop/orchestration" candidate from this series' own list.

**Package built today:**
- **AgentLoopKit** — a bounded, host-driven ReAct loop (`AgentLoop` actor) over `ProviderGatewayKit.LLMSession` + `ToolRegistryKit.ToolRegistry`: a pluggable `AgentResponseParser` turns raw model text into an `AgentDecision` (tool call vs. final answer — the bundled `JSONToolCallParser` recognizes `{"tool": ..., "arguments": {...}}`, everything else is the final answer verbatim), dispatched tool calls go through `ToolRegistryKit` for real schema validation, and a pluggable `AgentPromptStrategy` folds the result back into the next turn. Every run returns a complete `AgentTranscript` with a typed `AgentHaltReason` (`.finalAnswer` / `.maxStepsExceeded` / `.toolDispatchFailed` / `.parseFailed`) — `run(initialPrompt:)` only throws for a genuine session/router transport failure, never for "the agent didn't converge." 21 tests, 100.00% line/function/region coverage on every file, 0 SwiftLint violations (tool-verified).
  https://github.com/rajatslakhina/agent-loop-kit

**Design note — why this doesn't duplicate `ProviderRouter`'s built-in tool-calling:** `ProviderGatewayKit.ProviderRouter` already transparently drives *provider-native* tool-calling round trips when a provider's capabilities say `supportsToolCalling: true` (against its own minimal, string-only, unvalidated built-in `ToolRegistry`). `AgentLoopKit` is for the other, equally common case — exactly this ecosystem's own simulated/scripted providers — where the model only emits plain text and "decided to call a tool" has to be inferred from it, with the call itself going through `ToolRegistryKit` for real `JSONSchema` validation. It also generalizes the exact hand-rolled loop `runToolCallingScenario` in the shared demo was already writing out by hand for a *single* tool call — `AgentLoopKit` makes that a reusable actor that can chain multiple dependent tool calls.

**Coverage-gate design fix worth flagging:** hit a genuine "impossible to reach 100% coverage" wall from a `precondition(maxSteps > 0, ...)` in `AgentLoop.init` — Swift's `precondition` trap branch can't be exercised without crashing the test process, and a fresh coverage run of `foundation-model-provider-gateway` itself confirmed this is a pre-existing, silent, never-flagged gap across the whole series (`ProviderRouter.swift` sits at 90.48%, `LLMRequest.swift` at 75.00% — both solely due to their own `precondition` calls, despite the log never calling this out before today). Rather than repeat that pattern, redesigned `maxSteps` to clamp to `max(1, maxSteps)` instead of trapping — a real, testable, and arguably better public-API behavior — which is what let `AgentLoopKit` reach a literal, tool-verified 100.00%. Not fixed in the older packages today (out of scope for a light maintenance pass on other people's already-released public API), but worth a deliberate look in a future run with a larger time budget.

**Shared demo:** `llm-ecosystem-demo` extended with a sixth scenario — `AgentLoop.run(initialPrompt:)` chains two dependent `get_weather` calls (Austin, then Boston) before the model converges on a comparison answer, driven through the self-hosted provider identity (diversifying from the cloud identity the fifth, hand-rolled tool-calling scenario already uses). `TokenMeterKit` meters every step of the returned `AgentTranscript` entirely after the fact — `AgentLoopKit` never needs to know `TokenMeterKit` exists, demonstrating real package decoupling. Adding the sixth scenario pushed `EcosystemDemo.swift` over SwiftLint's file-length and type-body-length limits; split the new scenario into its own file (`AgentLoopScenario.swift`, an extension on `EcosystemDemo`) rather than suppressing the rule.
